# 方寸识途 (FloorRoute) 实操手册

本文档用于持续记录方寸识途 (FloorRoute) 的实际搭建、运行、验证和维护步骤。后续每次新增重要能力、改动项目结构、接入原生能力或后端接口，都应该继续更新这里。

## 1. 当前项目定位

方寸识途，英文名 FloorRoute，第一阶段是一个 mobile-first Web App 原型，后续通过 Capacitor 接入 Android Studio 打包成 App。

第一阶段目标：

```text
上传 / 拍摄平面图
  ↓
确认图片
  ↓
输入起点和终点
  ↓
模拟生成路径
  ↓
展示结果
  ↓
保存到历史
```

当前主 Tab：

```text
/nav        导航
/history    历史
/account    账号
```

## 7. Android 打包与签名说明

Android 打包分两种情况，不要混在一起：

### 7.1 测试打包

用于自己手机、模拟器、本地调试。

这种包通常是 debug 包，不需要手动准备发布密钥。Android Studio / Gradle 会使用本机 debug keystore。

本机已找到 debug keystore：

```text
C:\Users\xincy\.android\debug.keystore
```

测试打包常见方式：

```bash
npm run cap:sync:android
npm run cap:open:android
```

然后在 Android Studio 里运行 App，或者构建 debug APK。

也可以在命令行进入 Android 工程后构建：

```powershell
cd android
.\gradlew assembleDebug
```

debug 包不能用于正式发布到应用商店。

### 7.2 正式发布打包

用于上传 Google Play 或其他应用商店。

正式发布包需要 release keystore。这个 keystore 用来证明后续更新包和第一次发布的包来自同一个开发者。

如果是一个从未发布过的新 App，可以新建一个专属于 FloorRoute 的 release/upload keystore。

如果是已经发布过的旧 App，后续更新必须使用和旧版本兼容的签名密钥。密钥丢失会很麻烦：

- 如果启用了 Google Play App Signing，通常可以在 Play Console 里申请重置 upload key。
- 如果没有启用托管签名，又找不到原始 app signing key，通常无法继续更新同一个包名的 App，只能找回原密钥或换包名重新发布。

本机搜索到一个历史项目里的 keystore：

```text
C:\Users\xincy\Documents\Projects\rescue_auth_kit\android\app\upload-keystore.jks
```

这个看起来属于之前的 `rescue_auth_kit` 项目。不要随便拿它给 FloorRoute 复用，除非明确需要多个 App 使用同一签名身份。

### 7.3 FloorRoute 推荐做法

FloorRoute 目前还没有正式发布，因此建议后续新建一个独立 keystore，例如：

```text
C:\Users\xincy\.android\floor-route-upload.jks
```

推荐保存规则：

- keystore 文件不要提交到 Git。
- keystore 密码、key alias、key password 存到密码管理器。
- 可以在项目中使用 `keystore.properties` 记录本机路径和 alias，但这个文件必须加入 `.gitignore`。
- 在本文档里只记录 keystore 的保存位置和用途，不记录密码。

`.gitignore` 已加入以下规则，避免误提交签名文件：

```text
*.jks
*.keystore
*.p12
*.pfx
keystore.properties
```

正式打包推荐用 Android Studio：

```text
Build
  -> Generate Signed Bundle / APK
  -> Android App Bundle 或 APK
  -> 选择或新建 keystore
  -> 选择 release variant
```

如果是上传 Google Play，优先生成 AAB；如果只是发给别人安装测试，可以生成 APK。

### 7.4 这些 key 到底是什么

Android App 每个包都必须签名。签名不是“登录密码”，而是用来证明：

```text
这个安装包确实是同一个开发者/同一个项目后续生成的
```

几个概念：

`debug keystore`

本地调试用。Android Studio 自动生成，通常在：

```text
C:\Users\xincy\.android\debug.keystore
```

它只适合测试，不适合正式发布。

`release keystore`

正式发布用的密钥仓库文件，常见后缀是：

```text
.jks
.keystore
```

一个 keystore 里面可以放一个或多个 key。实际项目一般只放一个 key。

`key alias`

keystore 里面某一把 key 的名字。比如可以叫：

```text
floor-route-upload
```

`keystore password`

打开整个 keystore 文件的密码。

`key password`

使用某个 alias 对应的 key 进行签名时的密码。

为了少出错，个人项目可以让 `key password` 和 `keystore password` 相同，但必须足够强，并保存在密码管理器里。

`app signing key`

最终签到用户设备上、代表 App 身份的 key。

`upload key`

如果使用 Google Play App Signing，开发者本地通常用 upload key 给 AAB 签名，上传到 Play 后，Google 再用 app signing key 重新签发给用户。

简单理解：

```text
debug key       只管本地测试
upload key      你上传 Google Play 时证明“我是开发者”
app signing key 用户手机最终认可的 App 身份
```

### 7.7 项目里应该怎么引用 keystore

如果以后要配置命令行 release 打包，可以在本机创建一个不提交 Git 的文件：

```text
android/keystore.properties
```

示例字段：

```properties
storeFile=C:\\Users\\xincy\\.android\\floor-route-upload.jks
storePassword=这里写真实密码
keyAlias=floor-route-upload
keyPassword=这里写真实密码
```

注意：

- 这个文件包含真实密码，不能提交。
- 当前 `.gitignore` 已忽略 `keystore.properties`。
- 团队成员需要各自在自己电脑上配置自己的本地文件。

### 7.8 丢了 key 怎么办

如果只是丢了 debug keystore：

```text
没关系，可以重新生成。影响不大。
```

如果丢了 upload key，并且 App 使用 Google Play App Signing：

```text
可以在 Play Console 申请重置 upload key。
```

如果丢了自己管理的 app signing key，且没有启用 Google Play App Signing：

```text
通常无法继续给已发布 App 发更新，只能找回原 key，或者换包名重新发布。
```

因此正式发布前，保存 keystore 和密码这一步非常重要。
